SUPER-SAR Option Trading System

Production-Ready Multi-Threaded Algorithmic Trading System

Executive Summary

SUPER-SAR-Option is a robust algorithmic trading system designed for professional options trading on the Bombay Stock Exchange (BSE). The platform leverages a decoupled, multi-threaded architecture to deliver high-performance execution latency while maintaining enterprise-level reliability and compliance.

Key Capabilities:
  • Real-time multi-strike monitoring with automatic ATM selection
  • Proprietary algorithmic entry/exit signals with real-time validation
  • Paper trading & live execution with emergency kill-switch protection
  • Cross-market trend correlation via secure communications infrastructure
  • Multi-layered risk management and automated position protective protocols
  • Comprehensive audit logging and P&L tracking

System Architecture

The SUPER-SAR-Option platform is built on a modular, service-oriented architecture (SOA) that separates concerns across five distinct processing layers:

Level 1 Data Layer

WebSocket and REST API integration for real-time market data ingestion.

Level 2 Ingestion Layer

Dedicated sync engine for historical data reconciliation and state management.

Level 3 Strategy Engine

Technical indicator computation and multi-consensus signal generation.

Level 4 Risk Engine

Automated position management, stop-loss logic, and volatility filtering.

Level 5 Execution Engine

High-performance order placement and lifecycle management via broker APIs.

Level 6 Audit Layer

Structured logging, compliance tracking, and real-time P&L reporting.

Layer 1: Data Layer

The data layer handles all external data ingestion and acts as the primary interface to broker APIs.

  • FlatTrade REST API: Licensed broker integration for live market data and order execution
  • WebSocket Subscription: Real-time LTP (Last Traded Price) streaming for sub-second latency
  • Historical Data Retrieval: 3-day OHLCV candle data for technical analysis foundation
  • Quote Fetching: Periodic snapshot polls for VWAP, Ask/Bid prices, and Greeks

Layer 2: Ingestion Layer (Real-Time Syncer)

A dedicated historical_syncer() thread manages continuous data ingestion with intelligent caching and conflict avoidance.

Key Responsibilities:
  • Monitors global state for CE/PE token changes
  • Fetches 3-day historical bars every 30 seconds
  • Prevents "zero-value overwrites" during API hiccups
  • Calculates structural support/resistance from price peaks and valleys
  • Thread-safe state management via reader/writer locks

Layer 3: Strategy Engine

The strategy engine processes live data feeds to identify market opportunities based on proprietary quantitative models.

Layer 4: Risk Engine

The risk engine ensures capital preservation by enforcing per-trade and per-session risk limits, managing dynamic stop-loss levels, and validating market volatility conditions before entry.

Layer 5: Execution Engine (Order Manager)

The execution engine handles all order lifecycle management with thread-safety and rate-limit protection.

  • Session API Pooling: Maintains persistent broker session to avoid redundant logins
  • Order Pyramiding: Incremental lot scaling (Buy-1 → Buy-2 → Buy-3) on momentum confirmation
  • Emergency Kill Switch: Centralized emergency_exit() function cancels all pending orders and squares off positions in <2 seconds
  • Market Hours Enforcement: Trades only execute between 9:25 AM – 3:15 PM IST; auto square-off at 3:20 PM

Layer 6: Audit Layer (Logging & Compliance)

  • Structured Logging: JSON-formatted logs for all trades, signals, and errors
  • P&L Tracking: Real-time P&L aggregation across all completed trades
  • Trade Journal: CSV export of all trades with entry/exit rationale
  • System Health Monitoring: CPU/memory utilization tracking via psutil

Threading Model & Concurrency

Thread Safety Strategy:
  • RwLock Pattern: Reader/writer locks protect state variables (state.lock)
  • Immutable Data Sharing: DataFrame copies exchanged between threads
  • ThreadPoolExecutor: Bounded thread pool (max 4 workers) for order placement
  • Queue-Based IPC: thread.Queue for logging and async notifications
  • ZMQ Pub/Sub: Loose coupling between market trend publisher and subscriber threads

Signal Generation & Entry Logic

State Machine: Watch → Trade → Exit

IDLE STATE
↓ [ATR > threshold AND ST uptrend AND SAR bullish]
WATCH STATE (monitoring for entry)
↓ [LTP > HA_Close AND HA_Open = HA_Low AND ZMQ signal]
ENTRY EXECUTED (position opened)
↓ [Exit condition met: Target OR SL1 OR SL2]
POSITION CLOSED (trade completed)
↓ [Cooldown period (5 min)]
IDLE STATE (return to monitoring)

Detailed Entry Criteria (Pseudo-Code)

function check_watch_conditions(strike_data):
    atr = calculate_atr(strike_data.ha_df)
    st_val, st_dir = calculate_supertrend(strike_data.ha_df, 10, 1)
    sar_val, sar_dir = calculate_sar(strike_data.ha_df)
    
    condition_1 = atr > MIN_ATR_THRESHOLD  # ₹5-8
    condition_2 = st_dir == "UP"            # Supertrend bullish
    condition_3 = sar_dir == "UP"           # SAR bullish
    condition_4 = ltp >= res_level - 10     # Distance from resistance
    
    return condition_1 AND condition_2 AND condition_3 AND condition_4

function check_entry_signal(live_candle, watch_state):
    if NOT watch_state:
        return False
    
    ltp = live_candle.close
    ha_close = live_candle.ha_close
    ha_open = live_candle.ha_open
    ha_low = live_candle.ha_low
    
    condition_1 = ltp > ha_close           # Breakout
    condition_2 = (ha_open == ha_low)      # Flat-bottomed
    condition_3 = zmq_signal == BULLISH    # Cross-market validation
    
    return condition_1 AND condition_2 AND condition_3

Risk Management: Stop-Loss Dual Mode

SL1: ATR-Based (Volatility Stop)

Formula: SL = Entry Price - (1.5 × ATR)

Advantages:

  • Adapts to market volatility
  • Prevents whipsaws in trending markets
  • Ensures 1.5:1 risk/reward ratio

Use Case: High volatility (ATR > ₹10)

SL2: Structural (Support-Based)

Formula: SL = Low[T-1] (previous candle's low)

Advantages:

  • Aligned with actual support zones
  • Reduces false stops from noise
  • Better for range-bound markets

Use Case: Low volatility (ATR < ₹5)

Target Calculation

Dynamic Target Formula:
Target Price = Entry Price + (ATR / 3)

Example:
  • Entry: ₹100
  • ATR: ₹9
  • Target: ₹100 + (₹9 / 3) = ₹103
  • SL1: ₹100 - (1.5 × ₹9) = ₹86.50
  • Risk/Reward: ₹13.50 / ₹3 = 4.5:1
